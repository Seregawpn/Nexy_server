# Rapid Feature Integration Plan

## Метаданные
- Ассистент: codex
- Тип: analysis
- Дата: 2026-01-11
- ID (INS-###): INS-005

## Diagnosis
Новые фичи добавляются без стандартизированного «скелета» по архитектуре, из-за чего растет время интеграции и риск обхода EventBus/gateways.

## Root Cause
Нет закрепленного минимального пути «модуль + интеграция + контракт + конфиг» как единого источника истины → разработчики импровизируют → появляются дубли и нестабильность.

## Optimal Fix
Цель: формализовать минимальный архитектурный путь добавления фичи без новых сущностей, используя текущие слои и правила.

## Verification
DoD: есть краткий cookbook по шагам, привязанный к текущей архитектуре; все новые фичи проходят через EventBus, UnifiedConfig и gateways.

## Запрос/цель
Дать понятный способ быстрых (10–15 мин) добавлений новой фичи без крупных изменений.

## Контекст
- Документы: Docs/ARCHITECTURE_OVERVIEW.md, Docs/PROJECT_REQUIREMENTS.md, Docs/ARCHITECTURE_REQUIREMENTS_FINAL.md, Docs/ARCHITECTURE_FLAGS_AND_CONFIG_USAGE.md
- Ограничения: без реархитектуры, без нового центра управления

## Решения/выводы
- Фикс должен жить как «процедура» внутри текущей архитектуры: модуль → интеграция → EventBus контракт → конфиг/feature flags.
- Нужен единый минимальный шаблон действий, не новый кодовый фреймворк.
