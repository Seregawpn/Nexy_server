## 🧠 **ФИЛОСОФИЯ РАЗРАБОТКИ**

### **1. ПРИНЦИП "ДИАГНОСТИКА ПРЕЖДЕ ДЕЙСТВИЯ"**
> **"Сначала понять, потом исправлять"**

**Что мы изучили:**
- Всегда начинать с проверки логов и статуса сервисов
- Использовать `journalctl`, `systemctl status`, `curl` для диагностики
- Анализировать ошибки перед принятием решений

**Практическое применение:**
```bash
# Диагностика сервера
sudo journalctl -u service-name --no-pager -n 50
sudo systemctl status service-name
curl -s http://server/health
```

**Когда применять:** При любых проблемах с сервисами, ошибках, неработающем функционале.

---

### **2. ПРИНЦИП "СИСТЕМНОГО ПОДХОДА"**
> **"Видеть всю систему, а не отдельные компоненты"**

**Что мы изучили:**
- Сервер состоит из множества взаимосвязанных компонентов
- Проблема в одном месте может влиять на другие
- Нужно проверять все сервисы: HTTP, gRPC, Update, Database

**Практическое применение:**
```bash
# Проверка всех компонентов
curl http://server/health      # HTTP Server
curl http://server:8081/health # Update Server  
telnet server 50051           # gRPC Server
```

**Когда применять:** При деплое, обновлениях, диагностике проблем.

---

### **3. ПРИНЦИП "ПРАВ ДОСТУПА И БЕЗОПАСНОСТИ"**
> **"Права доступа - основа стабильной работы"**

**Что мы изучили:**
- Файлы должны принадлежать правильному пользователю
- Сервисы запускаются от имени root, но файлы могут принадлежать другим пользователям
- Права доступа критичны для работы Update сервера

**Практическое применение:**
```bash
# Исправление прав доступа
sudo chown -R root:root /path/to/files
sudo chmod -R 755 /path/to/files
```

**Когда применять:** При проблемах с чтением файлов, ошибках "Permission denied".

## 🌐 **ПРИНЦИПЫ ДЕПЛОЯ**

### **7. ПРИНЦИП "ДВУХУРОВНЕВОЙ СИНХРОНИЗАЦИИ"**

**Что мы изучили:**
- Локальные изменения нужно синхронизировать с GitHub
- GitHub автоматически обновляет production сервер
- Важно поддерживать консистентность между репозиториями

**Практическое применение:**
```bash

```

**Когда применять:** При любых изменениях в коде или конфигурации.

---

### **8. ПРИНЦИП "СЕТЕВОЙ ДОСТУПНОСТИ"**
> **"Все порты должны быть открыты на всех уровнях"**

**Что мы изучили:**
- Порт может быть открыт в UFW, но заблокирован в Azure NSG
- Нужно проверять firewall на VM и Network Security Groups в Azure
- Внешний доступ требует правильной настройки сети

**Практическое применение:**
```bash
# Проверка локального firewall
sudo ufw status
sudo ufw allow 8081/tcp

# Проверка Azure NSG
az network nsg rule list --resource-group RG --nsg-name NSG
```

**Когда применять:** При проблемах с доступом к сервисам извне.

---

### **9. ПРИНЦИП "ЛОГИРОВАНИЯ И МОНИТОРИНГА"**
> **"Логи - источник истины о состоянии системы"**

**Что мы изучили:**
- Логи показывают реальное состояние сервисов
- Важно читать логи в реальном времени при диагностике
- Разные уровни логов дают разную информацию

**Практическое применение:**
```bash
# Мониторинг в реальном времени
sudo journalctl -u service-name -f

# Анализ ошибок
sudo journalctl -u service-name --no-pager -n 100 | grep ERROR
```

**Когда применять:** При диагностике проблем, мониторинге работы сервисов.
