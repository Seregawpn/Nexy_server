## 📋 **ОБЗОР**

Этот документ содержит универсальные принципы и правила для безопасного и надежного деплоя приложений на любые серверы и в любые среды. Правила основаны на анализе реальных проблем и их решений.

---

## 🔧 **ПРИНЦИП 1: КОНФИГУРАЦИЯ И СЕКРЕТЫ**

### ✅ **ПРАВИЛА:**
- **Всегда проверять наличие конфигурационных файлов** на целевой среде
- **Секреты должны быть в переменных окружения**, не в коде
- **Использовать .env файлы** для локальной разработки
- **Документировать все необходимые переменные** окружения
- **Использовать разные конфигурации** для разных сред (dev/staging/prod)

### 🚨 **ЧТО МОЖЕТ ПРОИЗОЙТИ:**
- Сервисы не запускаются из-за отсутствия конфигурации
- API ключи не загружаются, модули не инициализируются
- Секреты попадают в репозиторий

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Проверить наличие config.env
ls -la config.env

# Проверить переменные окружения
env | grep -E "(API|KEY|SECRET)"

# Проверить загрузку конфигурации
grep -r "load_dotenv\|config" main.py
```

---

## 📁 **ПРИНЦИП 2: СТРУКТУРА ФАЙЛОВ И ИМЕНОВАНИЕ**

### ✅ **ПРАВИЛА:**
- **Следовать паттернам именования**, указанным в коде/документации
- **Проверять соответствие файлов** ожидаемым паттернам
- **Использовать версионирование** в именах файлов
- **Документировать требования** к структуре файлов
- **Проверять права доступа** к файлам перед запуском

### 🚨 **ЧТО МОЖЕТ ПРОИЗОЙТИ:**
- Сервисы не находят файлы из-за неправильного именования
- Обновления не работают из-за неправильных паттернов
- Права доступа блокируют чтение файлов

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Проверить паттерны именования
find . -name "manifest_*.json" -o -name "config_*.yaml"

# Проверить права доступа
ls -la updates/manifests/
ls -la config/

# Проверить соответствие коду
grep -r "glob.*manifest" modules/
```

---

## 🌐 **ПРИНЦИП 3: СРЕДОВАЯ КОНФИГУРАЦИЯ**

### ✅ **ПРАВИЛА:**
- **Никогда не использовать localhost** в production конфигурации
- **Все URL должны указывать** на правильные хосты для каждой среды
- **Создавать отдельные конфигурации** для каждой среды
- **Валидировать все URL и endpoints** перед деплоем
- **Использовать переменные окружения** для хостов и портов

### 🚨 **ЧТО МОЖЕТ ПРОИЗОЙТИ:**
- AppCast содержит localhost URLs
- Клиенты не могут подключиться к серверу
- Неправильные endpoints в production

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Проверить URL в конфигурации
grep -r "localhost\|127.0.0.1" config/
grep -r "http://" manifest*.json

# Проверить переменные окружения
echo $SERVER_HOST
echo $SERVER_PORT
```

---

## 🔐 **ПРИНЦИП 4: ПРАВА ДОСТУПА И БЕЗОПАСНОСТЬ**

### ✅ **ПРАВИЛА:**
- **Проверять права доступа к файлам** перед запуском сервисов
- **Документировать пользователя** запуска сервисов
- **Использовать принцип минимальных привилегий**
- **Регулярно проверять права** к критическим файлам
- **Использовать sudo только когда необходимо**

### 🚨 **ЧТО МОЖЕТ ПРОИЗОЙТИ:**
- Сервисы не могут читать конфигурационные файлы
- Update сервер не находит manifest файлы
- Permission denied ошибки

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Проверить права доступа
ls -la updates/manifests/
ls -la config/

# Проверить пользователя сервиса
systemctl show voice-assistant.service | grep User

# Исправить права
sudo chown -R root:root updates/
sudo chmod -R 755 updates/
```

---

## 🌐 **ПРИНЦИП 5: СЕТЕВАЯ БЕЗОПАСНОСТЬ**

### ✅ **ПРАВИЛА:**
- **Всегда проверять открытые порты** в firewall
- **Документировать все необходимые порты**
- **Проверять сетевые правила** для новых сервисов
- **Тестировать доступность всех endpoints** после деплоя
- **Использовать HTTPS в production**

### 🚨 **ЧТО МОЖЕТ ПРОИЗОЙТИ:**
- Сервисы недоступны извне
- Update сервер не отвечает
- gRPC порт заблокирован

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Проверить открытые порты
ss -tlnp | grep -E ":(8080|8081|50051)"
ufw status

# Проверить доступность
curl -s http://server:8080/health
curl -s http://server:8081/health

# Проверить Azure NSG
az network nsg rule list --resource-group RG --nsg-name NSG
```

## 🚀 **ПРИНЦИП 6: ДЕПЛОЙ И CI/CD**

### ✅ **ПРАВИЛА:**
- **Всегда иметь план ручного деплоя** как fallback
- **Тестировать деплой в staging** перед production
- **Документировать все шаги деплоя**
- **Проверять работоспособность всех сервисов** после деплоя
- **Использовать версионирование** для всех артефактов

### 🚨 **ЧТО МОЖЕТ ПРОИЗОЙТИ:**
- GitHub Actions не работает
- Автоматический деплой падает
- Нет способа быстро откатиться

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Проверить статус CI/CD
gh run list --limit 5

# Ручной деплой
git pull origin main
systemctl restart service

# Проверить версию
git log --oneline -1
```

## 📊 **ПРИНЦИП 7: МОНИТОРИНГ И ДИАГНОСТИКА**

### ✅ **ПРАВИЛА:**
- **Всегда проверять логи сервисов** при проблемах
- **Иметь health check endpoints** для всех сервисов
- **Мониторить статус всех компонентов** системы
- **Документировать процедуры диагностики**
- **Использовать централизованное логирование**

### 🚨 **ЧТО МОЖЕТ ПРОИЗОЙТИ:**
- Проблемы не обнаруживаются вовремя
- Нет информации для диагностики
- Сервисы падают без уведомления

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Проверить логи
journalctl -u service --no-pager -n 50

# Проверить health endpoints
curl -s http://server:8080/health
curl -s http://server:8081/health

# Проверить статус сервисов
systemctl status service
```

## ⏱️ **ПРИНЦИП 8: ТАЙМАУТЫ И НАДЕЖНОСТЬ**

### ✅ **ПРАВИЛА:**
- **Всегда устанавливать таймауты** для внешних операций
- **Использовать неблокирующие команды** где возможно
- **Иметь fallback варианты** для критических операций
- **Мониторить время выполнения** операций
- **Использовать retry механизмы** с exponential backoff

### 🚨 **ЧТО МОЖЕТ ПРОИЗОЙТИ:**
- Команды зависают без завершения
- Долгие операции блокируют процесс
- Нет способа прервать зависшие операции

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Использовать таймауты
timeout 30 command

# Проверить время выполнения
time command

# Использовать неблокирующие команды
command &
```

- **Тестировать обновления зависимостей** в staging

- Конфликты версий библиотек
- Несовместимые зависимости
- Проблемы с protobuf/grpcio

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Проверить версии зависимостей
pip list | grep -E "(grpcio|protobuf)"

# Проверить requirements.txt
cat requirements.txt

# Обновить зависимости
pip install -r requirements.txt
```

## 🔄 **ПРИНЦИП 10: ROLLBACK И ВОССТАНОВЛЕНИЕ**

### ✅ **ПРАВИЛА:**
- **Всегда иметь план быстрого отката**
- **Автоматически сохранять предыдущие версии** конфигурации
- **Тестировать процедуры rollback**
- **Иметь backup критических данных**
- **Документировать процедуры восстановления**

### 🚨 **ЧТО МОЖЕТ ПРОИЗОЙТИ:**
- Нет способа быстро откатиться при проблемах
- Потеря критических данных
- Долгое время восстановления

### 🛠️ **ПРИМЕРЫ ПРОВЕРОК:**
```bash
# Создать backup
cp -r config/ config.backup.$(date +%Y%m%d_%H%M%S)/

# Откатиться к предыдущей версии
git reset --hard HEAD~1

# Восстановить из backup
cp -r config.backup.*/ config/
```

## 📋 **УНИВЕРСАЛЬНЫЕ ШАБЛОНЫ**

### **Шаблон проверки конфигурации:**
```bash
check_config() {
    echo "🔍 Checking configuration..."
    
    # Проверить наличие конфигурационных файлов
    if [ ! -f "config.env" ]; then
        echo "❌ config.env not found"
        return 1
    fi
    
    # Проверить переменные окружения
    if [ -z "$API_KEY" ]; then
        echo "❌ API_KEY not set"
        return 1
    fi
    
    # Валидировать URL
    if grep -q "localhost" config.env; then
        echo "❌ localhost found in production config"
        return 1
    fi
    
    echo "✅ Configuration check passed"
    return 0
}
```

### **Шаблон проверки прав доступа:**
```bash
check_permissions() {
    echo "🔐 Checking permissions..."
    
    # Проверить права к файлам
    if [ ! -r "config.env" ]; then
        echo "❌ Cannot read config.env"
        return 1
    fi
    
    # Проверить права к директориям
    if [ ! -r "updates/manifests/" ]; then
        echo "❌ Cannot read manifests directory"
        return 1
    fi
    
    echo "✅ Permissions check passed"
    return 0
}
```

### **Шаблон проверки сети:**
```bash
check_network() {
    echo "🌐 Checking network..."
    
    # Проверить открытые порты
    if ! ss -tlnp | grep -q ":8080"; then
        echo "❌ Port 8080 not listening"
        return 1
    fi
    
    # Проверить доступность endpoints
    if ! curl -s http://localhost:8080/health > /dev/null; then
        echo "❌ Health endpoint not responding"
        return 1
    fi
    
    echo "✅ Network check passed"
    return 0
}
```

### **Шаблон деплоя:**
```bash
deploy() {
    echo "🚀 Starting deployment..."
    
    # 1. Подготовка
    check_config || exit 1
    check_permissions || exit 1
    check_network || exit 1
    
    # 2. Backup
    create_backup
    
    # 3. Деплой
    deploy_application
    
    # 4. Проверка
    check_services
    run_smoke_tests
    
    # 5. Уведомление
    notify_team
    
    echo "✅ Deployment completed successfully"
}
```
