## 📊 **Текущая ситуация**

### **1. Проблема: HTTP 502 Bad Gateway**

**Симптомы:**
- Публичный endpoint `https://20.151.51.172/health` возвращает **502 Bad Gateway**
- Nginx не может подключиться к внутренним сервисам (порт 8080 или 50051)
- Проблема на стороне сервера/инфраструктуры (не protobuf)

**Архитектура доступа:**
- **Внешний доступ:** Только через Nginx на порту 443 (HTTPS)
- **Внутренние порты:** 8080 (HTTP health), 50051 (gRPC), 8081 (Updates) — только localhost
- **Nginx проксирует:** `https://20.151.51.172/health` → `http://127.0.0.1:8080/health`
- **Nginx проксирует:** `https://20.151.51.172/` (gRPC) → `grpc://127.0.0.1:50051`

**Возможные причины:**
1. Сервис не запущен или упал (внутренние порты не слушают)
2. Сервер перегружен и не может обработать запрос
3. Проблема с Nginx конфигурацией (неправильные upstream)
4. Сетевая проблема на localhost

---

### **2. Проблема: Azure CLI заблокирован**

**Симптомы:**
- Ошибка: `ERROR: (Conflict) Run command extension execution is in progress`
- Невозможно выполнить новые команды на VM
- Предыдущая команда зависла или выполняется очень долго

**Причина:**
- Azure CLI позволяет выполнять только **одну команду одновременно** на одной VM
- Предыдущая команда все еще выполняется (возможно, долгая операция)
- Новые команды блокируются до завершения предыдущей

**Проблемные команды:**
- Долгие операции (`sleep`, `curl` без таймаута)
- Длинные скрипты, превышающие таймаут Azure CLI (~90 секунд)
- Команды без таймаутов для сетевых операций

## 🔍 **Что нужно проверить**

### **1. Статус systemd сервиса**

```bash
sudo systemctl status voice-assistant.service
```

**Ожидаемый результат:**
- Статус: `active (running)`
- Процесс Python запущен
- Нет ошибок в логах

**Если сервис неактивен:**
```bash
sudo systemctl restart voice-assistant.service
```

---

### **2. Внутренние порты (localhost)**

```bash
ss -tlnp | grep -E ':(50051|8080|8081)'
```

**Ожидаемый результат:**
- Порт 8080 слушает на `127.0.0.1:8080` или `0.0.0.0:8080`
- Порт 50051 слушает на `[::]:50051` или `0.0.0.0:50051`
- Порт 8081 слушает на `127.0.0.1:8081` или `0.0.0.0:8081`

**Если порты не слушают:**
- Сервер не запущен или упал
- Перезапустите сервис: `sudo systemctl restart voice-assistant.service`

---

### **3. Health endpoint (локально)**

```bash
curl -s http://127.0.0.1:8080/health
```

**Ожидаемый результат:**
```json
{
  "status": "ok",
  "latest_version": "1.0.0",
  "latest_build": "1"
}
```

**Если endpoint не отвечает:**
- Сервер не запущен или упал
- Проверьте логи: `sudo journalctl -u voice-assistant.service -n 30`

---

### **4. Логи сервиса**

```bash
sudo journalctl -u voice-assistant.service -n 30
```

**Что искать:**
- Ошибки инициализации
- Ошибки подключения к базе данных
- Ошибки загрузки модулей
- Критические ошибки

---

### **5. Конфигурация Nginx**

```bash
sudo nginx -t
cat /etc/nginx/sites-enabled/nexy-grpc | grep -E '(upstream|127.0.0.1)'
```

**Ожидаемый результат:**
- Nginx конфигурация валидна
- `upstream grpc_backend` указывает на `127.0.0.1:50051`
- `upstream http_backend` указывает на `127.0.0.1:8080`

## 🔧 **Что нужно сделать**

### **Шаг 1: Разблокировать Azure CLI**

**Вариант 1: Подождать завершения команды**
- Подождать 10-15 минут
- Проверить, что команда завершилась
- Выполнить новую команду

az vm restart --resource-group Nexy --name nexy-regular

**Вариант 3: Использовать SSH напрямую**
```bash

```

---

### **Шаг 2: Исправить проблему 502**

**1. Перезапустить сервис:**
```bash
sudo systemctl restart voice-assistant.service
```

**2. Проверить статус:**
```bash
sudo systemctl status voice-assistant.service
```

**3. Проверить health endpoint:**
```bash
curl -s http://127.0.0.1:8080/health
```

**4. Перезагрузить Nginx:**
```bash
sudo systemctl reload nginx
```

**5. Проверить публичный endpoint:**
```bash
curl -sk https://20.151.51.172/health
```
